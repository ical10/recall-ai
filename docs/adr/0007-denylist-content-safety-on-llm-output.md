# Denylist-based content safety on LLM output

CLAUDE.md lists content-safety checks among the pydantic validators on every LLM output, but the initial `SimpleVocabExample` schema only enforced shape, length, and the "example contains token" rule. We add a `_no_disallowed_terms` validator that scans `definition` and `example` for word-boundary matches against a small denylist of slurs and explicit terms. The denylist itself is intentionally **not** checked into git — it's the kind of content that should not live in public diff history. The repo ships an empty tuple in `app/services/content_safety.py`; populate it locally from a gitignored override or env-driven loader before going live. Failures plug into the existing retry-with-prompt-refinement loop in `LLMClient.complete`, so the model gets up to three chances to regenerate clean output; persistent failures land in the per-item cooldown described in ADR-0005.

Considered (a) a second LLM call to classify content — rejected as doubling per-item token cost for a v1 with a single trusted user; (b) outsourcing safety entirely to the model's own RLHF — rejected because dev defaults to `z-ai/glm-4.5-air:free`, whose safety filtering is weaker than commercial models and the rule should hold regardless of which provider/model is configured. The denylist is starting point, not the ceiling: refine over time as failures surface in the worker logs (`content_gen_item_failed`).
