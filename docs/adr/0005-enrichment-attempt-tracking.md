# Enrichment attempt tracking with cooldown

A nightly Celery task enriches Vocab Items whose `definition` is still empty. Without bookkeeping, a permanently-bad item (typo, non-word token, unsupported language) would be retried every night forever — burning ~600 tokens per item per night across the LLM's three internal retries. We add two columns to `vocab_items`: `enrichment_attempts: int` (default 0) and `last_enrichment_attempted_at: timestamptz | None`. The worker increments on each failed attempt and resets to 0 on success. `select_unenriched` skips items with `enrichment_attempts >= 3` whose last attempt is within `COOLDOWN_DAYS` (7) — so a broken row goes quiet after 3 nights and re-tries weekly.

Considered (a) a separate `enrichment_failures` table for per-attempt audit — rejected as over-built for v1; (b) reusing the empty-string sentinel with a second magic value (e.g., `"__failed__"`) — rejected because it mixes user-visible content with operational state and undermines the simplicity of ADR-0001. The chosen approach scales to inspect-and-fix-by-hand operations: `SELECT token, enrichment_attempts FROM vocab_items WHERE definition='' AND enrichment_attempts >= 3` shows what's stuck, and `UPDATE ... SET enrichment_attempts=0 WHERE id=...` forces a retry on the next cron tick.
